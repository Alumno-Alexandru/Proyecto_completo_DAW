BRIEFING - PROYECTO_COMPLETO_GITGAME_STORE
Resumen del proyecto

Proyecto_completo-main es una versión didáctica de una tienda de videojuegos (GitGame Store) que reúne un frontend estático y módulos backend de ejemplo. El objetivo es demostrar la construcción de una interfaz de catálogo, carrito de compras básico y un sistema de suscripciones con lógica del lado cliente y ejemplos de manejo en Python.

Alcance y componentes clave (breve)

- Frontend estático en `html/`: páginas principales (index, catálogo, favoritos, ofertas, suscripciones).
- Estilos con Tailwind y CSS personalizados en `src/css/`.
- Interactividad cliente en `src/javascript/`: carrito (`app.js`), suscripciones (`subscriptions.js`), navegación y utilidades.
- Datos de ejemplo en `src/data/` (`db.json`, `suscripciones.json`).
- Módulos Python en `src/python/` con un gestor de suscripciones y pruebas unitarias (`subscriptions_manager.py`, `api_subscriptions.py`, `test_subscriptions.py`).
- Scripts y manifestos básicos (`package.json`, `pnpm-lock.yaml`) para flujo frontend si se usa build tooling.

Estado actual

- Interfaz completa y navegable desde archivos estáticos (puedes abrir `html/index.html`).
- Sistema de suscripciones funcional: 4 planes, toggle mensual/anual y cancelación con modal.
- Lógica de cambios de plan implementada en cliente; la parte de pago real fue retirada para simplificar (no hay integración real de pago incluida).
- Backend en Python es de ejemplo y no persiste en base de datos por defecto (usa estructuras en memoria/JSON de ejemplo).

Instrucciones rápidas (uso y pruebas)

- Abrir la web: abrir `html/index.html` en el navegador.
- Probar suscripciones: `html/suscripciones.html` — cambiar entre mensual/anual y usar "Actualizar Ahora" o "Cancelar Suscripción".
- Ejecutar pruebas Python (recomendado en virtualenv):

```bash
python -m venv .venv
# Windows
.venv\Scripts\activate
pip install -r src/python/requirements.txt
python -m pytest src/python/test_subscriptions.py
```

Limitaciones conocidas (resumen)

- No hay integración real de pasarela de pago (solo lógica cliente y simulaciones retiradas).
- No existe autenticación de usuarios ni vinculación real de suscripciones a cuentas persistentes.
- Persistencia es de ejemplo (JSON/localStorage); no hay base de datos configurada.
- APIs Python son de ejemplo y requieren adaptación para producción (seguridad, validaciones, autenticación).

Mejoras recomendadas (priorizadas)

1. Autenticación y usuarios: integrar un sistema de cuentas (Flask-Login / JWT) para asociar suscripciones.
2. Persistencia: añadir base de datos (SQLite/Postgres) y migraciones para guardar usuarios, pedidos y suscripciones.
3. Pasarela de pago: integrar Stripe o PayPal para pagos reales, con webhooks de verificación.
4. Separación API/Frontend: convertir el backend de ejemplo en una API REST/GraphQL y desacoplar frontend para consumo.
5. Tests y CI: ampliar tests unitarios y añadir pipeline CI (GitHub Actions) para ejecutar linters y tests.
6. Internacionalización y accesibilidad: textos multi-idioma y mejoras ARIA para accesibilidad.
7. Documentación técnica: añadir diagramas y guías de despliegue (Docker, despliegue en nube).

Siguientes pasos sugeridos (rápidos)

- Añadir autenticación básica y persistencia en una rama separada.
- Integrar y probar Stripe en modo sandbox si se requiere pagos.
